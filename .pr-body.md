## Summary

Closes the supply-chain shadowing gap that the abandoned PR #974 (Tod Hansmann / todpunk) was trying to fix on the hex tarball download path. PR #1209 already forward-ported the metadata side via `order_members_local_first`; this PR completes the pair by adding the matching guard on the bytes side (`GET /hex/{repo}/tarballs/{name}-{version}.tar`).

The bug: on a virtual hex repo with members ordered `[hex-remote (priority 1), hex-local (priority 2)]`, an attacker who pushes a package named `phoenix` to the remote upstream can shadow the operator's locally-published `phoenix` because `resolve_virtual_download` iterates members in configured priority order. The metadata endpoint catches this by reordering members local-first regardless of configured priority; the tarball endpoint did not.

What changed:

- `package_name_from_tarball_filename` (new pure helper) parses `<name>-<version>.tar` into the name. Hex package names may contain dashes (`my-package-1.4.0.tar`), so the split is "first `-` followed by an ASCII digit." Matches the parser shipped on PR #974.
- `virtual_local_owns_tarball_name` (new) returns true if any non-Remote member of a virtual hex repo has an artifact row matching the parsed package name (case-insensitive via the existing `proxy_helpers::find_artifact_by_name_lowercase`).
- `serve_virtual_tarball_local_only` (new) calls `proxy_helpers::resolve_virtual_download` with `proxy_service: None`, so the iteration cannot escape to an upstream proxy fetch even if member priority orders Remote first.
- `download_tarball`: when the guard fires (local member owns the name), routes through the local-only path. Otherwise the existing `try_remote_or_virtual_download` path handles the request unchanged.
